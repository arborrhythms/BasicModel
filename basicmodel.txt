BasicModel algorithm overview
=============================

This file describes the syntax-enabled model selected by
data/BasicModel.xml. It records the current decisions explicitly so that a
future experiment does not turn an implementation detail into an accidental
architectural rule.

Current dimensions
------------------

  PartSpace WHAT/event width                 P = 128 / 136
  WholeSpace WHAT/event width                H = 128 / 136
  ConceptualSpace WHAT/event width           C = 1024 / 1032
  SS concept-code activation width               1024 / 1032
  InputSpace word packing buckets            W = 16, 32, 64, 128
  maximum sentence words in one invocation  Wmax = 128
  live PartSpace locations                   8
  live WholeSpace locations                  8
  live ConceptualSpace result locations      8
  ShortTermMemory capacity                   K = 8 by default (parameterized)
  part codebook rows                         32768 allocated, 1048576 ceiling
  conceptual codebook rows                   1048576 reserved, 32768 active
                                               (one table shared by all folds)
  initial WholeSpace property rows            8
  subsymbolicOrder                            T = 4

The widths above name different axes. In particular, W is neither a PS/WS/CS
field width nor the STM capacity, and neither codebook cardinality is a live
field width. PS and WS are intentionally compact, independent perceptual
peers. SS uses sparse concept-code activation. ConceptualSpace stays
high-dimensional, and the conceptual and whole-percept inventories are
independent.


1. InputSpace owns the word loop
---------------------------------

Before the compiled recurrence, InputSpace counts the actual surface words in
the batch and selects the smallest bucket which fits them. It then stages a
rectangular W-position carrier. An eager sentence adapter walks that carrier
in order and replays one shared, compiled K=2 word cell exactly W/2 times.
Padding positions are tensor-masked no-ops. The adapter copies each two-word
slice into a canonical layout, so all four packing widths and dynamic residual
part widths reuse the same compiled recurrence rather than producing four
whole-sentence graphs. STM and the other recurrent state are returned from one
cell invocation to the next, preserving strict serial word order.

A sentence longer than 128 words currently selects W=128, processes the first
128 words, and sets the sentence-truncation diagnostic. Splitting such a
sentence is not yet implemented. This sentence bound must not be confused
with truncating a long word: within an admitted sentence, all residual
constituents of each complete word are staged. Their local P axis is dynamic
and is independent of PartSpace.nOutput=8.


2. Canonical word decomposition and PS synthesis
-------------------------------------------------

The canonical BasicModel path is synthesis=mereology. The currently available
discrete implementation uses the mereological radix store as follows:

  1. InputSpace advances to word w and presents that word's complete local
     part ids, mask, and byte locations to PartSpace.
  2. The radix store spells the word from left to right by greedy longest-known
     prefix. If no longer known prefix starts at a location, a one-byte
     percept is the bottom case.
  3. A word-bounded chunk of at least chunkPromotionMinLength bytes (2 by
     default) seen chunkPromotionThreshold times (2 in BasicModel.xml) is
     promoted to one stored percept, initialized from the mean of its previous
     spell-out. An already stored chunk is not counted again. Thus familiar
     words usually leave very few residual percepts, often one, while an
     unfamiliar word may still have several.
  4. Only after PS receives the local part state does
     PartSpace.synthesize_word_parts gather all residual events and form the
     parameter-free De Morgan M-to-1 mereological base P0. The learned sigma
     rungs begin only after P0, so sigma[0] is the first order-raising fold,
     not part of the base aggregation. No raw constituent is discarded merely
     because the live PS field is eight positions wide.

The radix rows and P0 are direct membership vectors in [0,1]^128. For
constituents m_i, the base operation is 1-product_i(1-m_i), with 0 as the
sigma identity. It is many-to-one and therefore is not claimed to be
invertible; reconstruction uses the retained constituent ids and fold-order
record. In particular, there is no signed chart at this boundary: applying
(1+x)/2 here would turn two absent percepts into positive evidence.

Steps 2 and 3 are the present radix/chunk policy, not a claim that this is the
final mereological combination law. Moving promotion or part/whole
combination into the subsymbolic loop remains a legitimate experiment. The
important invariant is already enforced: InputSpace owns traversal, PS owns
synthesis, and a complete word produces one base part state.

BPE, MPHF, lexicon, byte/none, and the old explicitly named radix front ends
are compatibility paths selected and dispatched through bin/Legacy.py. They
do not participate in the synthesis=mereology branch. The canonical branch
still uses the radix data structure internally because it is presently the
implementation of the mereological store; this does not make the legacy
"radix" configuration mode a second live synthesis law.


3. Subsymbolic folds and actual concept order
----------------------------------------------

With subsymbolicOrder T=4, one word iteration has a base field followed by
three cumulative folds on each tower:

  P1 = sigma[0](P0)       H1 = pi[0](H0)
  P2 = sigma[1](P1)       H2 = pi[1](H1)
  P3 = sigma[2](P2)       H3 = pi[2](H2)

P0..P3 and H0..H3 remain in their respective 128-D native WHAT spaces. Each
learned unary sigma/pi rung is an invertible membership-domain map
[0,1]^128 -> [0,1]^128. It uses a bounded monotone coupling butterfly: two
endpoint-safe shears around an endpoint-fixing scalar curve, all with
closed-form inverses. It does not use tanh/atanh activation squashing, a
log-membership chart, or the signed truth chart. Exact 0/1 endpoints remain
finite in both directions and in their adjoints. (A bounded tanh
parameterization is used only for the learned coupling coefficients.)

When a fold contributes to a concept, the RMS of its native WHAT coordinates
is already a scalar in [0,1]. That scalar selects and scales a signed unit row
of the shared conceptual codebook. Reading the row emits the 1024-D conceptual
WHAT value in [-1,1]; this indexed dictionary read, not sigma itself, is the
dimension increase. CS receives a 1032-D event and does not widen a 136-D
event. P0's RMS(union), H0's RMS(property state), and the six learned PS/WS
values (plus a valid prior-STM peer when present) are
averaged, which preserves the signed bound. Updated dictionary rows are
reprojected locally to unit length after the optimizer step, using only the
rows named by the sparse gradient rather than scanning the million-row table.

The ordered path is stored, not inferred from the number of outputs. A strict
aligned fold raises if an advertised sigma or pi rung is absent or cannot
consume its input; an identity/no-fit rung is not silently counted as
evidence.

The word concept draws simultaneously from P0's RMS(union), all three
part-fold activations, H0's RMS(property state), and all three whole-fold
activations. ConceptualSpace retains the eight-source carrier and its ordered
fold-support map. The SS peer is the complete
prior-tick STM slab: STM slot i supplies concept row i and its scalar
activation to CS location i through the same indexed conceptual-codebook read.
It is decoded before the current word is pushed, so no same-tick CS->SS->CS
shortcut exists. An empty or unknown SS location is absent rather than a zero
ninth vote: the local mean is over 8+valid_SS sources and the eight perceptual
sources are not diluted where SS has no exact row identity.

The concept's actual subsymbolic order is stored explicitly as the deepest
contributing perceptual fold (3 in this configuration), while the support map
records that orders 1, 2, and 3 all contributed from both perceptual sides.
Exactly one word concept, not three or eight concepts, enters STM. Its exact row
and lexical coefficient are stored beside the dense STM payload for later SS
recursion. A unary or genuinely binary grammatical transform produces a dense
composite with no single exact row, so its row becomes unknown and its SS
activation becomes zero. A no-op preserves the reference, and an explicit
hard COPY may preserve the selected operand's reference. This metadata is
honest lexical provenance, not a claim that every dense composite has already
been sparsely factorized.

This actual concept order is not called "order 0", and it is not inferred
from a codebook row range. Raw percept order, cumulative fold depth,
mereological rank, part of speech, and grammatical derivation depth are
related empirical quantities but are not aliases. The current carrier records
fold order and concept order; a future explicit per-percept order can be added
without changing the STM contract.


4. Three-peer, location-aligned concept activation
---------------------------------------------------

PS and WS are two distinct native 128-D, eight-location perceptual subspaces.
SS supplies sparse symbolic activation over the conceptual codebook. They are
peers feeding CS, not a serial tower, and PS and WS are not one
undifferentiated sixteen-slot field:

  PS = [p0, p1, ..., p7]
  WS = [h0, h1, ..., h7]

On conceptBinding=aligned, only equal locations combine. At each recursion,
PS, WS, and SS advance independently and feed sparse activations of the same
concept-code namespace to CS. A PS or WS fold's scalar presence is measured
only in its own native space for a known joint concept identity. The
selected conceptual row is then read at 1024 WHAT dimensions; its trailing
eight where/when coordinates remain a separate aligned band. CS combines
already-conceptual-width activations at location n and retains each source and
fold order for provenance and reverse recovery.

There is no coordinate correspondence asserted between PS/WS WHAT dimensions
and CS WHAT dimensions, no fixed isometry, and no CS-owned feature lift. A
136-D PS event is never padded, repeated, or reshaped into a 1032-D concept.
The dimension increase occurs only by sparse conceptual codebook activation,
after the invertible native sigma/pi fold. An unknown concept row remains
unknown; it must not alias row zero or trigger a full million-row scan.

The dependency graph is

  {PS, WS, SS} --sparse sigma/codebook activation--> CS

not PS -> WS -> CS. The native peer computations may therefore be batched or
fused before their common CS synchronization point. A pipeline optimization
must preserve that graph; it must not make one perceptual peer consume the
other. In the serial word path the synthesized PS word occupies location 0,
so the resulting word concept entering STM is CS[0].

BasicModel explicitly disables the older whole-sentence ``sentenceProtocol``
prelude. That compatibility path still presents raw native-width PS/WS events
to a generic CS pad/trim routine, so enabling it would bypass the sparse
codebook boundary described above. It may be restored only after its gist is
formed from indexed conceptual activations rather than coordinate padding.

The same location rule is used by the aligned parallel path. The learned
ConceptualCombine mixing matrix is a separate, conditional parallel-only
alternative; BasicModel does not allocate or call it. It must not be used to
define serial alignment or to supply the codebook dimension increase.


5. Independent conceptual and whole-percept inventories
--------------------------------------------------------

ConceptualSpace.nVectors is the cardinality of the conceptual codebook. In
BasicModel it reserves 1048576 rows, with 32768 initially active. The four
aligned subsymbolic stages share that one physical table; stages do not carry
four learned copies of the same namespace. ConceptualSpace owns concepts, META
relations, and their taxonomy. SymbolSpace owns no duplicate concept rows: a
symbol is a sparse reference/activation of a ConceptualSpace concept id. That
activation is an SS peer input at the recursion boundary, while the referenced
identity and high-D code remain CS-owned; neither is stored in WholeSpace.

PartSpace has a separate 1048576-row logical ceiling. It initially allocates
only 32768 rows. New radix/chunk percepts are queued as detached CPU values
during a word pass, then installed at the post-backward/optimizer-step boundary
while the physical table grows geometrically. The boundary migration preserves
existing row ids and optimizer state, and invalidates compiled ownership before
the next call. Exhausting maxVectors is atomic: no partial row or trie update is
made. This avoids paying for a one-million-row PartSpace before the vocabulary
requires it.

The storage consequence is explicit. At fp32 and conceptual event width 1032,
the one shared 1048576-row ConceptualSpace parameter is 4.03125 GiB. The
canonical indexed-only dictionary does not allocate the unused VQ EMA mirror
or cluster counts, removing another 4.035 GiB; legacy VQ codebooks retain their
EMA state. Canonical aligned reads use sparse embedding gradients, and the
dedicated row-local Adam keeps two power-of-two prefix moment slabs in BF16,
while performing each touched-row update in fp32. At the initial 32768-row
prefix, W, the two moments, the norm cache, and active mask total about
4.162 GiB. Even at full occupancy that core is about 8.067 GiB, rather than
about 20.16 GiB for W, an EMA mirror, a dense gradient, and two fp32 Adam
moments. A normal word tick's sparse gradient contains only the indexed rows it
touched, and exact-zero placeholder rows are discarded before optimizer state
is changed. Keeping CS high-dimensional still dominates memory, but its
million-row address space no longer implies three extra full fp32 copies.

The initial 32768x128 PartSpace parameter is 16 MiB. If PS eventually fills
its 1048576-row ceiling, that parameter is 512 MiB; stock Adam adds 1 GiB of
moment tensors and a dense gradient adds another 512 MiB (2 GiB total for
parameter, moments, and gradient). WS begins with only eight 128-D property
rows, so its persistent inventory is negligible. Geometric PS growth defers
the endpoint. A row-local PS optimizer remains a useful follow-up if occupancy
grows that far.

The PS128/WS128/CS1024 dimensional cutover starts from data/BasicModel.ckpt,
beside data/BasicModel.xml. An old checkpoint is not implicitly prefix-loaded
or partially reshaped; any future migration must be an explicit, separately
tested operation.

WholeSpace.nVectors has a different meaning: it is the capacity of the
whole-percept/property codebook. It starts with roughly eight ASCII-class
properties. It may later grow as dynamic properties are learned, potentially
toward PartSpace scale, but that growth is independent of conceptual capacity,
symbol count, STM capacity, and WholeSpace.nOutput. WholeSpace contains no
concept, META, taxonomy, or symbol-prototype rows and has no second
concept-sized analysis codebook. Property rows are independent predicates and
may overlap: a capital activates both LETTER and CAPITAL without allocating a
new conjunction row.

For aligned serial processing the conceptual dictionary is one fixed,
unpartitioned namespace. Concepts of different actual orders may address the
same dictionary; their order travels as metadata. The old snap/pool or
per-order row partitions remain relevant only to their separate
sparse-parallel code path and do not partition this model's serial conceptual
inventory.

Concept ids and their learned rows are stable identities and are never
recycled. In particular, retiring a transient concept does not make its id or
row available to a different definition. Reuse means exact identity reuse
through the existing word, relation, chain, and synthesis keys; it never means
overwriting the least-recently-used learned row.

A new word's A=WORD, B=OBJECT, and C=META concepts are one atomic three-id
admission. The allocator checks room for all three and reveals the required
active prefix before changing any relation table, so a two-row remainder
cannot leave a half-built word. Explicit concept-construction APIs retain the
same fail-loud rule for all grouped structures, including grammatical chains:
if the complete allocation cannot fit, no member of it is minted.

Evidence-driven automatic discovery has a different endpoint policy so a
long training run can continue safely at the physical ceiling. Known exact
identities remain live and continue accruing evidence. An unseen automatic
word, location, sentence-chain candidate, or higher-order promotion remains a
transient tensor/STM value instead of acquiring a durable id. The model logs
the first refusal in each class and accumulates checkpointed admission-drop
counts, exposed by ConceptualSpace.concept_admission_stats(). A rejected word
also suppresses its presentation-local location fallback, so one or two spare
rows cannot bypass the atomic A/B/C gate. No learned row, EMA state, sparse
edge, optimizer state, or stale symbolic reference is silently reassigned.

"Aligned" refers only to the live location axis: PS fold location n binds with
WS fold location n when forming a concept. It never asserts equal PS, WS, CS,
or SymbolSpace inventory sizes. Conceptual reserve growth and WholeSpace
property growth therefore have separate allocation and checkpoint policies.

Checkpoint schema 2 records this ownership explicitly. On first load of an
older checkpoint, the legacy concept/symbol-valued WholeSpace `.what` tensors
and the duplicate `analysis_store` tensors are omitted rather than prefix-loaded
as properties. Their Python taxonomy sidecars are quarantined for an explicit
ConceptualSpace import; the newly initialized property rows remain intact.


6. STM and the per-word grammar budget
--------------------------------------

STM is a parameterized bounded workspace with default capacity K=8. Every
active word iteration inserts exactly one word concept. Concept order and
grammar order are stored in parallel metadata slabs:

  - insertion copies the word concept's actual subsymbolic order and starts
    grammatical derivation depth at 0;
  - a binary grammatical operator consumes two values and returns one,
    reducing live STM depth by exactly one. The parent's concept order is the
    maximum of its children; its grammatical depth is one plus the maximum
    child grammatical depth;
  - a unary grammatical operator consumes and replaces the top value without
    changing STM depth. It preserves concept order and increments grammatical
    depth when it actually applies.

There is no separate cognitive operation named SHIFT here: insertion is the
last part of the word iteration. REDUCE refers only to an actual licensed
binary arity-2 grammar operation returning one value.

Each word has a strict hot-loop budget of at most one actual binary operation
and then at most one unary operation. If insertion would overflow STM, the
binary opportunity occurs before insertion as a demand; that row does not get
a second binary operation afterward. Otherwise the binary decision follows
insertion. The unary decision then runs on the surviving top value. COPY/no-op
router choices do not masquerade as applications.

Below capacity, grammar confidence is compared with the occupancy-dependent
threshold

  threshold(d) = tau0 * (1 - clip((d - 2) / (K - 2), 0, 1)),

where tau0=0.75 in BasicModel. Pressure is soft at low occupancy and becomes a
demand at capacity. If no valid binary operation exists when space is
required, the model raises rather than dropping a value or substituting an
unlicensed mean. Sentence-boundary forest closure is separate from the
per-word budget and may perform the additional binary operations required to
produce the sentence result.

In serialObjectMeta mode the grammar/STM sentence result is the composition
path; it replaces automatic durable sentence chains. The earlier
create_joint_concept call built an additional right-branching Gallistel chain
for nearly every presented sentence, consuming roughly one conceptual row per
word even though the serial grammar had already reduced those words. That
automatic sentence-sized inventory is now suppressed. The explicit chain API
and the nonserial compatibility path remain available where a caller
intentionally requests a named sequence type.


7. MPS benchmark (2026-07-19)
------------------------------

These measurements are the PS504/WS504/CS504 baseline from before the
PS128/WS128/CS1024 dimensional cutover. They remain useful for comparison, but
they are not a post-change speed claim; rerun the same benchmark after the
next fresh MPS model starts.

bin/bench_word_buckets.py benchmarks the real BasicModel forward after eager
surface/radix staging, including every masked word iteration, PS/WS fold,
aligned binding, and bounded STM grammar step. On this Apple MPS machine,
batch=1, one warm-up and three measured eager iterations gave:

  bucket      median seconds      effective words/second
  W=16            0.8638                    18.52
  W=32            1.7848                    17.93
  W=64            3.6116                    17.72
  W=128           7.2497                    17.66

The near-linear time above W=16 confirms that bucket width, rather than STM
depth, controls this loop's work. The JSON record is
output/bench_word_buckets_mps_eager.json.

The MPS-default torch.compile backend (backend="eager", fullgraph=True) was
then captured once per distinct fixed-W code object. Each post-capture sample
was run with Dynamo's error_on_recompile enabled. W=16 was measured three
times; the other widths use one post-capture sample because their first traces
are expensive. No same-bucket call retraced, including the warm-to-promoted
change in the dynamic residual-percept axis:

  bucket   first call seconds   steady seconds   words/second   vs eager
  W=16                 87.36           0.2875          55.65       3.00x
  W=32                175.62           0.5079          63.01       3.51x
  W=64                343.44           1.0830          59.09       3.33x
  W=128               683.43           2.1723          58.92       3.34x

Here "first call" includes Dynamo graph capture plus that call's execution; it
is not steady-state latency. The backend removes Python-loop overhead but does
not use Inductor kernel fusion on MPS. The measured graph should therefore be
cached and reused within a process; eagerly capturing every bucket at startup
would be counterproductive. The compiled JSON records are
output/bench_word_buckets_mps_compile_eager_w16_final.json,
output/bench_word_buckets_mps_compile_eager_w16_w32.json,
output/bench_word_buckets_mps_compile_eager_w64.json, and
output/bench_word_buckets_mps_compile_eager_w128.json.
